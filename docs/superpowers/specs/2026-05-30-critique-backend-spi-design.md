# Critique Backend — Qhorus Channel Design Spec

**Date:** 2026-05-30 (revised 2026-05-31)

## What Changed From First Draft

The original spec designed a `ReviewerBackend` HTTP-RPC SPI with two implementations
(Claude direct + LangChain4j). That was the wrong architecture for three reasons:

1. DraftHouse's declared dependency is `casehub-qhorus` — the reviewer belongs in the
   channel mesh, not behind a REST endpoint.
2. Architecture decision D2 (research spec) already chose LangChain4j. Dual backends
   reopen that without justification.
3. DraftHouse's explicit goal is to exercise and harden the Qhorus normative layer —
   speech acts, commitments, deontic accountability. An HTTP-RPC model bypasses all of it.

This spec replaces the SPI with a Qhorus channel participant model and commits to
LangChain4j as the single LLM backend.

---

## Core Model

The reviewer is not an HTTP service. It is a **Qhorus channel participant**: a CDI
`MessageObserver` bean that listens on a per-session APPEND channel, receives QUERY
messages, calls a LangChain4j `@AiService`, and dispatches RESPONSE messages back.
Document context lives in Qhorus `SharedData`. `CritiqueResource` is replaced by the
MCP tool surface.

```
start_review MCP tool
  → creates Qhorus channel: drafthouse/{sessionId}   (APPEND semantic)
  → stores doc-a, doc-b in SharedData               (keyed by session)
  → registers ReviewerAgent instance in Qhorus
  → records session in ReviewSessionRegistry         (channelId → sessionId)

User (via MCP tool: send_message, type=QUERY)
  → correlationId auto-generated by Qhorus
  → Commitment created: reviewer obligated to RESPOND

ReviewerAgent  (@ApplicationScoped MessageObserver)
  → receives MessageReceivedEvent from InProcessMessageBus
  → filters: channelId in active sessions AND type == QUERY
  → loads doc-a, doc-b from DataService by SharedData key
  → calls DocumentReviewer @AiService (LangChain4j)
  → dispatches RESPONSE on same channel, inReplyTo + correlationId
  → Commitment automatically resolved by MessageService.dispatch()

end_review MCP tool
  → releases SharedData artefact claims
  → deregisters reviewer instance
  → removes session from ReviewSessionRegistry
  → optionally deletes channel (force teardown order)
```

---

## Normative Layer — Why This Matters

Every QUERY–RESPONSE pair exercises the Qhorus 4-layer accountability framework:

| Layer | What DraftHouse exercises |
|---|---|
| Illocutionary | QUERY = "I am asking"; RESPONSE = "I am answering"; DECLINE = "I cannot"; FAILURE = "it broke" |
| Commitment | QUERY creates an open Commitment on the reviewer; RESPONSE/DECLINE/FAILURE closes it |
| Temporal | `MessageDispatch.deadline` propagates to `Message.deadline`; Watchdog enforcement available via `ReactiveWatchdogService` |
| Enforcement | Commitment outcomes feed casehub-engine when that dependency is wired |

DraftHouse is the reference consumer that proves this framework is usable end-to-end.
The forward path to work items: when a reviewer identifies something requiring human
decision, a COMMAND to an oversight channel triggers casehub-work (future dependency).

## Qhorus Integration

### Channel naming

```
drafthouse/{sessionId}          — APPEND, open writers (Phase 2: no ACL)
```

APPEND semantic gives ordered, cursor-paginated message history — right for a
conversation thread. No `allowed_writers` restriction in Phase 2; add ACL when
multi-user sessions arrive.

### SharedData keys

```
drafthouse/{sessionId}/doc-a    — document A content (full text)
drafthouse/{sessionId}/doc-b    — document B content (full text)
```

Loaded once at `start_review`. Reviewer calls `DataService.getByKey()` per QUERY.
ArtefactClaims held for the session lifetime — prevents GC during active review.
Released at `end_review`.

### Message flow

```
User → send_message(channel, type=QUERY, content="...", deadline=PT5M)
       correlationId auto-generated by Qhorus
       Commitment: OPEN, obligor=reviewer, deadline=now+5m

ReviewerAgent → MessageReceivedEvent fires (InProcessMessageBus.fireAsync)
             → load doc-a, doc-b from DataService
             → call DocumentReviewer.review(docA, docB, selectionContext, query)
             → dispatch RESPONSE(inReplyTo=queryId, correlationId=..., content=...)
               Commitment: FULFILLED

On LLM failure:
             → dispatch FAILURE(inReplyTo=queryId, correlationId=..., content=reason)
               Commitment: FAILED

On out-of-scope query:
             → dispatch DECLINE(inReplyTo=queryId, correlationId=..., content=reason)
               Commitment: DECLINED
```

## ReviewerAgent

```java
@ApplicationScoped
public class ReviewerAgent implements MessageObserver {

    @Inject ReviewSessionRegistry sessions;
    @Inject DataService dataService;
    @Inject DocumentReviewer llm;
    @Inject MessageService messageService;

    @Override
    public void onMessage(MessageReceivedEvent event) {
        Message msg = event.message();
        if (msg.messageType() != MessageType.QUERY) return;

        ReviewSession session = sessions.find(msg.channelId());
        if (session == null) return;

        String docA = dataService.getByKey(session.docAKey()).content();
        String docB = dataService.getByKey(session.docBKey()).content();

        try {
            String response = llm.review(
                session.personality(), docA, docB,
                selectionContext(session), msg.content()
            );
            messageService.dispatch(MessageDispatch.builder(
                msg.channelId(), REVIEWER_INSTANCE_ID, MessageType.RESPONSE, response)
                .inReplyTo(msg.id())
                .correlationId(msg.correlationId())
                .build());
        } catch (Exception e) {
            messageService.dispatch(MessageDispatch.builder(
                msg.channelId(), REVIEWER_INSTANCE_ID, MessageType.FAILURE,
                "Reviewer error: " + e.getMessage())
                .inReplyTo(msg.id())
                .correlationId(msg.correlationId())
                .build());
        }
    }
}
```

`ReviewerAgent` is session-agnostic — it handles all active sessions. The
`ReviewSessionRegistry` is the routing table: `channelId → ReviewSession`.

`ReviewSession` record:
```java
public record ReviewSession(
    UUID channelId,
    String sessionId,
    String docAKey,
    String docBKey,
    DocumentSide selectionSide,   // enum: A | B
    String selectionText,         // nullable
    String personality            // from config, not from client
) {}

public enum DocumentSide { A, B }
```

`selectionSide` and `selectionText` are updated via `update_selection` MCP tool when
the user moves their cursor or changes their selection. Not per-QUERY — the registry
holds the current selection state for the session.

## LangChain4j @AiService

```java
@RegisterAiService
public interface DocumentReviewer {

    @SystemMessage("You are a document reviewer. {personality}")
    @UserMessage("""
        Document A (original):
        {documentA}

        Document B (revised):
        {documentB}

        {selectionContext}

        User query: {query}
        """)
    String review(
        @V("personality") String personality,
        @V("documentA") String documentA,
        @V("documentB") String documentB,
        @V("selectionContext") String selectionContext,
        @V("query") String query
    );
}
```

`selectionContext` is an empty string when no selection is active. When active:
`"Selected text (" + side + " side): " + selectionText`. The reviewer constructs
this from `ReviewSession` state before calling — not passed raw from client.

**Personality** is from `application.properties` in Phase 2. Not client-supplied.
Prompt injection risk from a client-controlled system prompt string is real and
avoided by design.

```properties
casehub.drafthouse.reviewer.personality=You review document changes for clarity, \
  accuracy, and coherence. Be specific and constructive.
```

**Provider** configured entirely via `application.properties`. Swapping provider
requires only config and dependency changes — no Java changes:

```properties
# Anthropic direct
quarkus.langchain4j.anthropic.api-key=${ANTHROPIC_API_KEY}
quarkus.langchain4j.anthropic.chat-model.model-name=claude-sonnet-4-6

# Vertex AI
quarkus.langchain4j.anthropic.vertex-ai.project-id=${ANTHROPIC_VERTEX_PROJECT_ID}
quarkus.langchain4j.anthropic.vertex-ai.location=${CLOUD_ML_REGION}
```

## MCP Tool Surface (replaces CritiqueResource)

| Tool | Action |
|---|---|
| `start_review` | Create channel, store docs in SharedData, register agent, return sessionId |
| `update_selection` | Update `ReviewSession.selectionSide` + `selectionText` in registry |
| `end_review` | Release artefact claims, deregister instance, remove session |

These are DraftHouse's own `@Tool` methods on a `DraftHouseMcpTools` class — not Qhorus
tools. They use Qhorus services (`ChannelService`, `DataService`, `InstanceService`,
`MessageService`) via CDI injection.

The external LLM (Claude Code or any MCP client) drives the review by calling
`start_review`, then Qhorus's own `send_message` tool to post QUERY messages, then
`check_messages` or `get_replies` to read RESPONSE messages.

`CritiqueResource` (`POST /api/critique`) is deleted. The 501 stub is gone.

## Module Structure

DraftHouse adopts the `api/` + `app/` hexagonal split when Qhorus is wired — per the
deep dive: "Will adopt api/ + app/ hexagonal structure when the first CaseHub foundation
dependency is wired in."

```
server/
  api/       — ReviewSession, DocumentSide, ReviewSessionRegistry (interface), no JPA
  app/       — ReviewerAgent, DocumentReviewer, DraftHouseMcpTools, ReviewSessionRegistryImpl
```

`api/` has no Qhorus dependency. `app/` depends on `casehub-qhorus` and `quarkus-langchain4j`.

## Error Contract

`ReviewerAgent.onMessage()` catches all exceptions and dispatches FAILURE. This ensures:
- Every QUERY gets a terminal message (RESPONSE, DECLINE, or FAILURE)
- Every Commitment is closed — no open obligations leak across sessions
- `CritiqueResource` / MCP callers can `get_replies` and always get a result

FAILURE content is a human-readable reason string. Rate limit errors, LLM auth
failures, and context-window overflows are all caught and dispatched as FAILURE.

## Testing

**Unit — `ReviewerAgent`:**
`casehub-qhorus-testing` module provides `InMemory*Store` implementations — no
database, no Qhorus JVM required. Inject a real `MessageService` backed by in-memory
stores. Inject a mock `DocumentReviewer`. Assert that a QUERY dispatch triggers a
RESPONSE dispatch with correct `inReplyTo` and `correlationId`. Test FAILURE path
by having the mock throw.

**Unit — `DocumentReviewer`:**
LangChain4j provides `@QuarkusTest` mock provider via test profile. Assert that
the system prompt contains personality, user prompt contains doc-a, doc-b, and
query. No real LLM call in unit tests.

**Integration — end-to-end:**
Full `@QuarkusTest` with a real H2 `qhorus` datasource, real Qhorus services, and a
stub `DocumentReviewer` that echoes input. Verify the full QUERY → Commitment → RESPONSE
→ Commitment-FULFILLED lifecycle via `CommitmentStore.findByCorrelationId()`. This
proves the normative layer is wired correctly — not just that the reviewer returns text.

No Playwright E2E for critique in Phase 2 — the UI panel remains a placeholder stub.

## Out of Scope (Phase 2)

- Streaming RESPONSE (Qhorus messages are atomic; streaming via Multi<String> is a
  future channel backend concern)
- 3-channel normative layout (`NormativeChannelLayout` is a Claudony SPI; not yet
  in DraftHouse)
- Watchdog deadline alerting (ReactiveWatchdogService exists; integrate when the
  reactive stack is active in DraftHouse)
- Selection-scoped per-selection channels (Phase 5 in the deep dive)
- Multi-LLM reviewers / personality library (Phase 6)
- casehub-work integration (future dependency; commitment outcomes feed it when wired)
