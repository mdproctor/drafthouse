# Critique Backend — Qhorus Channel Design Spec

**Date:** 2026-05-30 (revised 2026-05-31)

## What Changed From First Draft

The original spec designed a `ReviewerBackend` HTTP-RPC SPI with two implementations
(Claude direct + LangChain4j). That was the wrong architecture for three reasons:

1. DraftHouse's declared dependency is `casehub-qhorus` — the reviewer belongs in the
   channel mesh, not behind a REST endpoint.
2. Architecture decision D2 (research spec) already chose LangChain4j. Dual backends
   reopen that without justification.
3. DraftHouse's explicit goal is to exercise and harden the Qhorus normative layer —
   speech acts, commitments, deontic accountability. An HTTP-RPC model bypasses all of it.

This spec replaces the SPI with a Qhorus channel participant model and commits to
LangChain4j as the single LLM backend.

---

## Core Model

The reviewer is not an HTTP service. It is a Qhorus **ChannelBackend** — a per-session
backend implementation registered when a review session's channel is initialised, that
receives messages via `post()`, calls a LangChain4j `@AiService`, and dispatches RESPONSE
or DECLINE messages back. Document context lives in Qhorus `SharedData`. `CritiqueResource`
is replaced by the MCP tool surface.

```
start_review MCP tool (DraftHouse tool on /mcp)
  → creates Qhorus channel: drafthouse/{sessionId}   (APPEND semantic)
  → stores doc-a, doc-b in SharedData               (keyed by session)
  → registers per-session Qhorus instance            (InstanceService.register)
  → ReviewerChannelBackendFactory observes ChannelInitialisedEvent
    and registers ReviewerChannelBackend for this channel

User (via Qhorus tool on same /mcp: send_message, type=QUERY)
  → correlationId auto-generated by Qhorus
  → Commitment created: reviewer obligated to RESPOND or DECLINE

ReviewerChannelBackend.post(Message msg)
  → filters: type == QUERY
  → loads doc-a, doc-b from DataService by SharedData key
  → calls DocumentReviewer @AiService (LangChain4j)
  → ReviewResult.declined=false → dispatch RESPONSE (inReplyTo, correlationId)
    Commitment fulfilled
  → ReviewResult.declined=true  → dispatch DECLINE  (inReplyTo, correlationId)
    Commitment cancelled by refusal

On exception:
  → dispatch DECLINE with sanitized reason string   (not FAILURE — FAILURE discharges
    COMMAND obligations only; QUERY obligations are discharged by RESPONSE or DECLINE)
    Commitment cancelled

end_review MCP tool
  → releases SharedData artefact claims
  → deregisters Qhorus instance
  → removes session state
  → optionally deletes channel (force teardown order)
```

**MCP endpoint composition:** DraftHouse embeds `casehub-qhorus` as a Quarkus extension.
`QhorusMcpTools` is `@ApplicationScoped` and active by default. `quarkus-mcp-server-http`
discovers all `@Tool` methods across all CDI beans in the application context — DraftHouse
tools (`DraftHouseMcpTools`) and Qhorus tools (`QhorusMcpTools`) are both available on
DraftHouse's single `/mcp` endpoint. The external LLM connects once and can call
`start_review`, then Qhorus's `send_message`, `check_messages`, and `get_replies` directly.

## Normative Layer — Why This Matters

Every QUERY–RESPONSE pair exercises the Qhorus 4-layer accountability framework (ADR-0005):

| Layer | What DraftHouse exercises |
|---|---|
| Illocutionary | QUERY = directive/epistemic; RESPONSE = assertive discharge; DECLINE = negative assertive (refusal or cannot) |
| Commitment | QUERY creates `C(reviewer→user, inform_result)`; RESPONSE fulfils it; DECLINE cancels it with secondary explain obligation |
| Temporal | `MessageDispatch.deadline` propagates to `Message.deadline`; QUERY expiry → auto-DECLINE (not auto-FAILURE) |
| Enforcement | Commitment outcomes feed casehub-engine when that dependency is wired |

**Speech-act correctness:** FAILURE discharges COMMAND obligations only (accepted but not
completed). QUERY obligations are discharged exclusively by RESPONSE (answered) or DECLINE
(refused/cannot). LLM errors are DECLINE — the reviewer cannot answer, which is semantically
accurate regardless of whether the cause is out-of-scope or a technical failure.

DraftHouse is the reference consumer that proves this framework is usable end-to-end.
The forward path to work items: when a reviewer flags something requiring human decision,
a COMMAND to an oversight channel triggers casehub-work (future dependency).

## Qhorus Integration

### Channel naming

```
drafthouse/{sessionId}          — APPEND, open writers (Phase 2: no ACL)
```

APPEND semantic gives ordered, cursor-paginated message history — right for a conversation
thread. `allowed_writers` ACL deferred to multi-user sessions.

### SharedData keys

```
drafthouse/{sessionId}/doc-a    — document A content (full text)
drafthouse/{sessionId}/doc-b    — document B content (full text)
```

Loaded once at `start_review`. Reviewer calls `DataService.getByKey()` per QUERY.
ArtefactClaims held for the session lifetime — prevents GC during active review.
Released at `end_review`.

**Document size:** At `start_review`, check document sizes against
`casehub.drafthouse.reviewer.max-doc-chars` (configurable). If either document exceeds
the limit, return an error immediately rather than discovering the problem at first query.

### Message flow

```
User → send_message(channel, type=QUERY, content="...", deadline=PT5M)
       correlationId auto-generated by Qhorus
       Commitment: OPEN, debtor=reviewer, deadline=now+5m

ReviewerChannelBackend.post(msg)
       → load doc-a, doc-b from DataService
       → call DocumentReviewer.review(...)
       → ReviewResult.declined=false:
           dispatch RESPONSE(inReplyTo=queryId, correlationId=..., content=answer)
           Commitment: FULFILLED
       → ReviewResult.declined=true (out-of-scope or LLM error):
           dispatch DECLINE(inReplyTo=queryId, correlationId=..., content=reason)
           Commitment: CANCELLED (secondary explain obligation created by Qhorus)
```

## ChannelBackend (PLATFORM.md guidance)

Per PLATFORM.md: "`ChannelBackend` is per-channel — use it when a consumer needs to act
on messages from a specific channel. `MessageObserver` is a global broadcast — use it for
cross-cutting concerns." The reviewer is session-scoped, not cross-cutting — `ChannelBackend`
is the correct SPI.

### Registration (ADR-0008 pattern)

**Registration pattern (verified against ClaudonyChannelBackend):** use deregister-then-register
for idempotency — `ChannelInitialisedEvent` fires on every `initChannel()` call (including
repeated calls for the same channel). Startup recovery fires for all persisted channels.

```java
@ApplicationScoped
public class ReviewerChannelBackendFactory implements ReviewSessionRegistry {

    @Inject ChannelGateway gateway;
    @Inject DataService dataService;
    @Inject MessageService messageService;
    @Inject DocumentReviewer llm;

    private final ConcurrentHashMap<UUID, ReviewSession> sessions = new ConcurrentHashMap<>();

    void onChannelInitialised(@Observes ChannelInitialisedEvent event) {
        if (!event.channelName().startsWith("drafthouse/")) return;
        ReviewSession session = sessions.get(event.channelId());
        if (session == null) return;  // startup recovery: no in-memory session, skip
        ReviewerChannelBackend backend = new ReviewerChannelBackend(
            session, dataService, messageService, llm);
        gateway.deregisterBackend(event.channelId(), ReviewerChannelBackend.BACKEND_ID);
        gateway.registerBackend(event.channelId(), backend, "agent");
    }

    // ReviewSessionRegistry implementation
    @Override public Optional<ReviewSession> find(UUID channelId) { ... }
    @Override public void put(ReviewSession session) { sessions.put(session.channelId(), session); }
    @Override public void remove(UUID channelId) { sessions.remove(channelId); }
    @Override public void updateSelection(UUID channelId, DocumentSide side, String text) { ... }
}
```

`ReviewerChannelBackendFactory` observes `ChannelInitialisedEvent` (fired on creation and
on startup recovery) and registers a `ReviewerChannelBackend` for every channel whose name
starts with `drafthouse/`. **Startup recovery gap (Phase 2 known limitation):** `ChannelInitialisedEvent` fires on
startup recovery for every persisted channel. `ReviewSession` data (SharedData keys,
instanceId, personality) lives only in-memory — after a JVM restart there is no session
in memory for persisted channels. The factory **skips registration** when no session
exists for a channelId at init time. The channel remains in Qhorus; QUERYs sit unhandled
until `start_review` is called again. This is acceptable for Phase 2. Persisting session
state to a DraftHouse-owned store is deferred to Phase 3.

### ReviewerChannelBackend

**API note (verified against qhorus source):** `ChannelBackend.post()` signature is
`post(ChannelRef channel, OutboundMessage message)`. `OutboundMessage.messageId` is a
delivery-scoped UUID (not the ledger Long). To obtain `inReplyTo` for the reply dispatch,
look up the original QUERY by `correlationId.toString()` via `MessageService.findByCorrelationId()`.
`OutboundMessage.correlationId` is UUID and must be converted to String for the dispatch.
`MessageDispatch.builder()` requires `.actorType()` — use `ActorType.AGENT`.

```java
public class ReviewerChannelBackend implements ChannelBackend {

    private static final String BACKEND_ID = "drafthouse-reviewer";

    private final ReviewSession session;
    private final DataService dataService;
    private final MessageService messageService;
    private final DocumentReviewer llm;

    @Override public String backendId() { return BACKEND_ID; }
    @Override public ActorType actorType() { return ActorType.AGENT; }
    @Override public void open(ChannelRef channel, Map<String, String> metadata) {}
    @Override public void close(ChannelRef channel) {}

    @Override
    public void post(ChannelRef channel, OutboundMessage message) {
        if (message.type() != MessageType.QUERY) return;

        // Resolve inReplyTo: OutboundMessage.messageId is delivery-scoped UUID,
        // not the ledger Long. Look up the QUERY by correlationId.
        Long inReplyTo = messageService
            .findByCorrelationId(message.correlationId().toString())
            .map(m -> m.id)
            .orElse(null);

        String docA = dataService.getByKey(session.docAKey())
            .map(d -> d.content).orElse("");
        String docB = dataService.getByKey(session.docBKey())
            .map(d -> d.content).orElse("");

        ReviewResult result;
        try {
            result = llm.review(session.personality(), docA, docB,
                selectionContext(session), message.content());
        } catch (Exception e) {
            result = ReviewResult.decline("Reviewer encountered an error.");
        }

        MessageType type = result.declined() ? MessageType.DECLINE : MessageType.RESPONSE;
        messageService.dispatch(MessageDispatch.builder()
            .channelId(channel.id())
            .sender(session.instanceId())
            .type(type)
            .content(result.content())
            .inReplyTo(inReplyTo)
            .correlationId(message.correlationId().toString())
            .actorType(ActorType.AGENT)
            .build());
    }
}
```

Exception messages are never passed to dispatch. Only sanitized, fixed strings reach the
channel — API keys, stack traces, and endpoint URLs stay out of the tamper-evident ledger.

**Exception specificity note:** The original spec showed three specific catch clauses
(RateLimitException, AuthenticationException, ContextLengthException). Whether LangChain4j
Anthropic throws these exact types depends on the version. A single `catch (Exception e)` with
a sanitized message is always safe. The specific catch clauses can be added if exception types
are confirmed against the actual LangChain4j Anthropic 1.9.1 source during TDD.

## ReviewSession

```java
public record ReviewSession(
    UUID channelId,
    String sessionId,
    String instanceId,       // per-session Qhorus instance ID (from InstanceService.register)
    String docAKey,          // SharedData key: drafthouse/{sessionId}/doc-a
    String docBKey,          // SharedData key: drafthouse/{sessionId}/doc-b
    DocumentSide selectionSide,   // enum: A | B — nullable when no selection
    String selectionText,         // nullable when no selection
    String personality            // from config at session start, not from client
) {}

public enum DocumentSide { A, B }
```

`selectionSide` and `selectionText` are updated in-place (immutable record replaced in
registry) via the `update_selection` MCP tool. When `ReviewerChannelBackend.post()` fires,
it snapshots the current `ReviewSession` from the factory. If `update_selection` races with
a QUERY, the snapshot reflects selection state at dispatch time — which is the correct
grounding for the query.

**Per-session identity:** `start_review` calls `InstanceService.register()` with a
synthetic instanceId (`drafthouse-reviewer-{sessionId}`). This instanceId is stored in
`ReviewSession` and used as the sender in all `MessageDispatch` calls. Every RESPONSE and
DECLINE in the ledger is attributed to the specific session that produced it. `end_review`
calls `InstanceService.deregister()`.

## LangChain4j @AiService

```java
@RegisterAiService
public interface DocumentReviewer {

    @SystemMessage("{{personality}}")
    @UserMessage("""
        Document A (original):
        {{documentA}}

        Document B (revised):
        {{documentB}}

        {{selectionContext}}

        User query: {{query}}

        If this query is outside the scope of document review (e.g. general knowledge,
        unrelated topics), respond with declined=true and explain why in content.
        Otherwise respond with declined=false and your review in content.
        """)
    ReviewResult review(String personality, String documentA,
                        String documentB, String selectionContext, String query);
}

public record ReviewResult(boolean declined, String content) {
    public static ReviewResult decline(String reason) {
        return new ReviewResult(true, reason);
    }
}
```

`@SystemMessage("{{personality}}")` uses LangChain4j's runtime dynamic variant — the
system prompt is resolved at call time from the `personality` parameter, not at compile
time. This makes zero behavioral difference for Phase 2 (personality is config-sourced)
but eliminates an interface refactor when the Phase 6 personality library arrives.

**Template variable syntax:** `{{paramName}}` (double-brace) is the Quarkus LangChain4j
`@RegisterAiService` convention for parameter substitution — verified against production
`@AiService` interfaces in `casehub-qhorus/examples/`. No `@V` annotations needed; method
parameter names map directly. Used consistently across `@SystemMessage` and `@UserMessage`.

**Implementation risk — structured output:** When `@AiService` returns a record type
(`ReviewResult`) rather than `String`, Quarkus LangChain4j activates its structured output
pipeline: JSON schema instructions are injected into the prompt and the response is parsed
as JSON. This is a different code path from `String` return. The natural-language format
instructions in `@UserMessage` may conflict with LangChain4j's injected schema instructions.
Mitigation: test `DocumentReviewer` against the mock provider as the first implementation
step, before any other session integration is wired.

**Personality** is from `application.properties` in Phase 2. Not client-supplied.

```properties
casehub.drafthouse.reviewer.personality=You review document changes for clarity, \
  accuracy, and coherence. Be specific and constructive.
casehub.drafthouse.reviewer.max-doc-chars=100000
```

**Provider** configured entirely via `application.properties`:

```properties
# Anthropic direct
quarkus.langchain4j.anthropic.api-key=${ANTHROPIC_API_KEY}
quarkus.langchain4j.anthropic.chat-model.model-name=claude-sonnet-4-6

# Vertex AI (current setup)
quarkus.langchain4j.anthropic.vertex-ai.project-id=${ANTHROPIC_VERTEX_PROJECT_ID}
quarkus.langchain4j.anthropic.vertex-ai.location=${CLOUD_ML_REGION}
```

## MCP Tool Surface (replaces CritiqueResource)

DraftHouse's own `@Tool` methods on `DraftHouseMcpTools`:

| Tool | Action |
|---|---|
| `start_review` | Check doc sizes; create channel; store docs in SharedData; register Qhorus instance; return sessionId. Channel init fires `ChannelInitialisedEvent` → backend factory registers `ReviewerChannelBackend` |
| `update_selection` | Replace `ReviewSession` in factory registry with updated `selectionSide` + `selectionText` |
| `end_review` | Release artefact claims; deregister Qhorus instance; remove session from factory |

The external LLM then uses Qhorus tools directly on the same `/mcp` endpoint:
`send_message` (type=QUERY), `check_messages` or `get_replies` to read responses.

`CritiqueResource` (`POST /api/critique`) is deleted.

## Session Lifecycle and Cleanup

Phase 2 known gap: if `end_review` is never called (MCP client disconnect, crash, timeout),
ArtefactClaims persist, the channel remains registered, and the session entry stays in the
factory registry. This is acceptable for Phase 2 (single-user, embedded tool) and will
accumulate slowly.

Planned mitigation (Phase 3): use a Qhorus Watchdog on the session channel to detect
inactivity and trigger `end_review` automatically. The `ReactiveWatchdogService` exists
and supports condition-based alerting; integration is deferred until the reactive stack
is active in DraftHouse.

## Module Structure

DraftHouse adopts the `api/` + `app/` hexagonal split when Qhorus is wired:

```
server/
  api/     — ReviewSession, DocumentSide, ReviewResult, ReviewSessionRegistry (interface)
             No Quarkus dependency; no JPA; no Qhorus
  runtime/ — ReviewerChannelBackend, ReviewerChannelBackendFactory
               (implements ReviewSessionRegistry — owns the ConcurrentHashMap<UUID, ReviewSession>)
             DocumentReviewer (@RegisterAiService), DraftHouseMcpTools
             DraftHouseConfig (@ConfigMapping — Quarkus type, cannot go in api/)
             Depends on casehub-qhorus, quarkus-langchain4j-anthropic
```

Note: the module is named `runtime/` (not `app/`) per the actual Maven structure.

`ReviewSessionRegistry` is a pure-Java interface in `api/` — no Qhorus types. It exposes
session lookup and selection update. `ReviewerChannelBackendFactory` implements it in
`app/`. `DraftHouseMcpTools` injects `ReviewSessionRegistry` (the interface), not the
factory directly. This keeps `DraftHouseMcpTools` free of backend implementation details
and the `api/` module free of Qhorus types.

## Channel Topology

Phase 2 uses a single APPEND channel per session (`drafthouse/{sessionId}`). The platform's
normative 3-channel layout (work/observe/oversight with `allowedTypes` enforcement, via the
`NormativeChannelLayout` Claudony SPI) is not applied in Phase 2.

Divergence is deliberate: the research spec envisions per-selection channels
(`sel/{hash}`, `observe`) in Phase 5. Phase 2 validates the basic QUERY→RESPONSE→Commitment
lifecycle on a single channel first. Alignment with the normative 3-channel layout is a
Phase 5 concern when selection-scoped channels arrive.

## Testing

**Integration — H2 configuration:**
`quarkus.datasource.qhorus.jdbc.url=jdbc:h2:mem:qhorus;MODE=PostgreSQL;DB_CLOSE_DELAY=-1`
(per platform `quarkus-test-database` protocol). Without `MODE=PostgreSQL`, Qhorus Flyway
migrations fail on H2.

**Unit — `ReviewerChannelBackend`:**
`casehub-qhorus-testing` provides `InMemory*Store` implementations. Inject a real
`MessageService` backed by in-memory stores. Inject a mock `DocumentReviewer` returning
`ReviewResult`. Assert that a QUERY dispatch triggers a RESPONSE dispatch with correct
`inReplyTo` and `correlationId`. Assert that a declined `ReviewResult` dispatches DECLINE.
Assert that each caught exception type dispatches DECLINE with the sanitized string (not
the raw exception message).

**Unit — `DocumentReviewer`:**
LangChain4j test profile with a mock provider. Assert system prompt contains personality;
assert user prompt contains documentA, documentB, query; assert `declined=false` for
in-scope queries, `declined=true` for out-of-scope. No real LLM call in unit tests.

**Integration — commitment lifecycle:**
Full `@QuarkusTest` with real H2 `qhorus` datasource, real Qhorus services, stub
`DocumentReviewer`. Verify the complete lifecycle:
1. `start_review` → channel created → `ReviewerChannelBackend` registered
2. `send_message(QUERY)` → Commitment OPEN in `CommitmentStore`
3. Backend calls stub → dispatches RESPONSE
4. `CommitmentStore.findByCorrelationId()` → Commitment FULFILLED

This proves the normative layer is wired correctly — not just that the reviewer returns text.

No Playwright E2E for critique in Phase 2 — the UI panel remains a placeholder stub.

## Context Fidelity (Phase 2 scope reduction)

The research spec (R3) envisions: selected text, which side (A or B), whether in a changed
region, and corresponding text on the other side. Phase 2 passes only the selected text and
side. Diff-aware context (whether the selection is in a changed region, counterpart text)
is deferred to Phase 5 when selection-scoped channels and the diff API are designed.

---

## Out of Scope (Phase 2)

- Streaming RESPONSE (Qhorus messages are atomic)
- Normative 3-channel layout (Phase 5)
- Watchdog session cleanup (Phase 3 — `ReactiveWatchdogService` exists, integration deferred)
- Selection-scoped per-selection channels (Phase 5)
- Multi-LLM reviewers / personality library (Phase 6)
- casehub-work integration (future; commitment outcomes feed it when wired)
- `ask_reviewer` wrapper tool — defeats the goal of exercising the Qhorus channel model directly
